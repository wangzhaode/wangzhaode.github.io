关于微软实习面试

Veröffentlicht am 2016-06-28   |   in 杂谈

从16年3月份，我开始在微软的实习招聘网站上投递了我的简历，选择的岗位是Software Development Engineer(Suzhou),具体的描述在这里。投递首先需要建立一个51job的账号登陆进去，之后选择志愿，填写简历之后就可以等待笔试通知了，基本上简历问题不大的话都可以通过建立初筛的。

等到了4月1日给了短信通知参加4月6日晚上的在线笔试。笔试是在hihoCoder上面进行的，总共有四道题目，对于非ACM队员来说难度还是不小的，不过好像一般能答对一道多一点就可以通过笔试了，笔试题目在hihoCoder这里。



 


在4月18日收到HR的邮件，告知通过了笔试，接下来要参加面试。可能由于距离比较远，直接安排使用Skype面试，面试时间是4月27日上午。我的面试一共有三轮，据说是如果前两轮面试官觉得很棒可以省去第三轮；如果前两轮面试官都觉得不好就没有机会了；如果前两轮面试官都感觉还差不多就要有第三轮面试。
第一轮面试官根据我的简历简单的问了一下，然后根据我的简历情况让我在Skype的共享白板上面写了一段程序模拟一下cache替换的LRU算法；这写完之后他又让我写一个求第n个斐波那契数的程序，然后让我给这个程序设计测试用例，正好刚刚学完软件测试，我就说了下等价类划划分和边界测试，然后设计了几个测试用例，第一轮面试就结束了。
紧接着开始第二轮面试，这个面试官对于简历上的项目问的稍微多一点，对于细节相对问的多了一些，不过问的也不是很难。接下来还是写代码，仍然是在那个共享白板上面写，这个题目是要求将两个有序数组合并成一个，这个题目很简单，采用尾差法时间复杂度为O(m+n),leetcode有原题，这里是地址，面试官在看代码时会看的很仔细，哪里有一行冗余的或者错误的他便会让你再仔细看看哪里还可以修改下，然后等改完还会问复杂度。

等过了第二轮，HR说先让我不要挂断 Skype，等待结果。大概过了十分钟左右，通知我还有一轮面试。结果第三位面试官是一位外国的工程师，而且他并不懂中文，所以我只好硬着头皮上了。首先他问了我一些关于项目的问题，我英语水平不是很高，说的不是很好，可能他也看出来了然后跟我说那咱们做题吧。首先他让我写一个有序二叉树合并的程序，这种问题之前写的比较多，所以直接就写了出来，他看了下感觉还可以就没有了问太多。接着又给我出了一道题目，判断一个数组中是不是有两个数相乘得到一个指定的数。这个题目我之前也在leetcode上面做过，很快就写出来了。这个面试官是要把白板上的程序直接运行的，我第一次写的有点快他告诉我编译有一个错误让我在看看，我仔细看了一遍改掉了错误。然后他问我时间空间复杂度，我写的是一个时间复杂度是0(n)，但是由于我用了HashMap导致空间是2n,他让我改一下，把空间空间使用改为n,我最后写出来了一个空间使用是n，时间复杂度是O(nlogn)的程序，然后第三轮面试就结束了。
大概到5月3日收到了HR的电话告知面试通过，如果参与暑期实习最晚的入职时间是7月12日，实习时间是三个月，如果无法实习可以参加夏令营，不过夏令营只有几天时间，所以我还是选择实习。


后来又接到通知，今年实习生同意安排到北京实习，申请苏州的转正仍然是去往苏州，所以突然我又要去北京实习了。而且可能由于大部分实习都去北京，所以本来统一安排住宿的也取消了，变成房补的形式自己租房。在之后还收到了HR发来的定制T-shirt的邮件，统计型号入职之后发放，这还是很赞的！

Git笔记

创建版本库
$git init

本地库添加、修改文件命令
$git add *** //***为文件名
$git commit -m "注释"

查看仓库当前状态
$git status

查看修改内容
$git diff

查看提交历史
$git log
$git log --pretty=onelin //只查看部分内容

回退到上一版本
$git reset --hard HEAD^ //HEAD表示当前版本，HEAD^表示上一版本，HEAD^^表示上上版本,HEAD~100表示往上100个版本
$git reset --hard commit_id //commit_id可以通过log查看得到，如果已关闭窗口可以用下面的命令

记录之前每次的命令
$git reflog

恢复修改、删除的内容
$git checkout --*** //***为文件名，此命令丢弃未add的修改
$git reset HEAD *** //***为文件名，此命令丢弃未commit的修改

删除文件
$git rm *** //***为文件名
$git commit -m "注释"

要关联一个远程库
$git remote add origin git@server_name:path/project_name.git
$git push origin master //第一次push的命令为 $git push -u origin master

从远程库克隆一个工程，以下两种方式皆可
$git clone git@server_name:path/project_name.git
$git clone https://server_name/path/project_name.git

操作分支命令 为分支名
$git branch //查看分支情况
$git branch //创建分支
$git checkout //切换分支
$git checkout -b //创建+切换分支
$git merge //合并某分支到当前分支
$git branch -d //删除指定分支
$git branch -D //强制删除指定分支

处理合并分支时的冲突（当两个分支都有改动时合并，无法自动合并）
$git merge //出现无法自动合并提示,手动修改后执行以下命令
$ git add *** //***为文件名
$ git commit -m "conflict fixed"

合并后保存分支历史的合并方式
$git merge --no-ff -m "注释"

首先，可以试图用git push origin branch-name推送自己的修改；

如果推送失败，则因为远程分支比你的本地更新，需要先用git pull试图合并；

如果合并有冲突，则解决冲突，并在本地提交；

没有冲突或者解决掉冲突后，再用git push origin branch-name推送就能成功！

如果git pull提示“no tracking information”，则说明本地分支和远程分支的链接关系没有创建，
用命令git branch --set-upstream branch-name origin/branch-name。

这就是多人协作的工作模式，一旦熟悉了，就非常简单。

Android事件传递机制

Veröffentlicht am 2014-11-18   |   in Android . 笔记

1.Android事件

归根结底，Android事件的基础只有三点

  • 按下（ACTION_DOWN）
  • 移动（ACTION_MOVE）
  • 抬起（ACTION_UP）
    其余动作诸如长按、滑动、双击等事件其实就是这三个事件的不同组合而已

2.事件消费者

Android的事件消费者可以分为两类

  • ViewGroup
  • View
    其实ViewGroup也是继承自View,不过ViewGroup是用来对View布局用的组件，所以其内部会嵌套View

例如常见的ViewGroup有各种Layout、ListView等
他们处理事件的方法是

ViewGroup有三个方法

public boolean dispatchTouchEvent(MotionEvent event)
public boolean onTouchEvent(MotionEvent event)
public boolean onInterceptTouchEvent(MotionEvent event)

View有两个方法

public boolean dispatchTouchEvent(MotionEvent event)
public boolean onTouchEvent(MotionEvent event)


1.其中dispatchTouchEvent方法用于事件的分发，返回true表示不继续分发，事件没有被消费。返回false则继续往下分发，如果是ViewGroup则分发给onInterceptTouchEvent进行判断是否拦截该事件，如果是View则交给onTouchEvent进行处理。

2.onInterceptTouchEvent是ViewGroup中才有的方法，View中没有，它的作用是负责事件的拦截，返回true的时候表示拦截当前事件，不继续往下分发，交给自身的onTouchEvent进行处理。返回false则不拦截，继续往下传。

3.onTouchEvent方法用于事件的处理，返回true表示消费处理当前事件，返回false则不处理，交给子控件进行继续分发。

 

所以可以看出事件的传递顺序为：

当TouchEvent发生时，首先Activity将TouchEvent传递给最顶层的View， TouchEvent最先到达最顶层 view 的 dispatchTouchEvent ，然后由 dispatchTouchEvent 方法进行分发，如果dispatchTouchEvent返回true ，则交给这个view的onTouchEvent处理，如果dispatchTouchEvent返回 false ，则交给这个 view 的 interceptTouchEvent 方法来决定是否要拦截这个事件，如果 interceptTouchEvent 返回 true ，也就是拦截掉了，则交给它的 onTouchEvent 来处理，如果 interceptTouchEvent 返回 false ，那么就传递给子 view ，由子 view 的 dispatchTouchEvent 再来开始这个事件的分发。如果事件传递到某一层的子 view 的 onTouchEvent 上了，这个方法返回了 false ，那么这个事件会从这个 view 往上传递，都是 onTouchEvent 来接收。而如果传递到最上面的 onTouchEvent 也返回 false 的话，这个事件就会“消失”，而且接收不到下一次事件。

3.事件消费与事件处理

不难看出，你可以在此流程中的任意一个方法中处理事件，并且同时允许事件继续分发而且不拦截，但是要记住，此时你仅仅是根据event进行某些操作而非消费了此事件，一个事件本质上只能被消费一次。那么同样的处理时间，消费与否有何区别。

其实，如果只是三个基本动作，处理起来没有什么区别。但是，双击、长按、拖动等动作事件其实是由多个基础事件构成的，此时只有你消费掉了第一个ACTION_DOWN你才能继续监听以后的动作以连贯起来构成一个复杂动作，而未消费的则无法生成复杂动作事件。

void cv2wx(IplImage ipl, wxImage &wx ) {
// Copy image data from Ipl (opencv) image to wxImage
// Assumes ipl image has seq “GBR”, depth 8, and 3 channels, and
// has the same size as the wxImage, and width_step = width3.
copy_and_swap_rb( ipl->imageData, (char)wx.GetData(),

IplImage cv_from_wx(wxImage &wx) {
// Return a new IplImage copied from a wxImage.
// Must be freed by user with cvReleaseImage().
IplImage ret = cvCreateImage(cvSize(wx.GetWidth(), wx.GetHeight()),
IPL_DEPTH_8U, 3);
wx2cv(wx, ret);
return ret;
}

wxImage wx_from_cv( IplImage cx) {
// Return new wxImage copied from a compatible IplImage.
// Assumes ipl image has seq “GBR”, depth 8, and 3 channels
// Fear not. The copy on return is cheap; does not deep-copy the data.

cv2wx(cx, wx);
return wx;

wxWidgets笔记 第二篇 分析HelloWorld

Veröffentlicht am 2014-10-28   |   in C++

分析第一段wxWidgets代码

看到自己复制代码并运行的HelloWorld之后，会不会感觉内容很陌生啊？
那么问题来了！

第一个问题：main函数在哪里？

首先你会看到这段代码
class MyApp : public wxApp
{
public:
virtual bool OnInit();
};
这是因为每一个wxWidgets程序都需要定义一个wxApp类的子类，并且需要创建并且只能创建一个这个类的实例
你的这个继承自wxApp的子类至少需要定义一个OnInit函数，当wxWidgets准备好运行你写的代码的时候，它将
会调用这个函数（这和一个典型的Win32程序中的main函数或者WinMain函数类似）。

第二个问题：OnInit如何定义？

看这下面段代码
bool MyApp::OnInit()
{
//MyFrame frame = new MyFrame(“Hello World”, wxPoint(50, 50), wxSize(450, 340));
MyFrame frame = new MyFrame(“Hello World”, wxDefaultPosition, wxDefaultSize);
frame->Show(true);
return true;
}
在这个OnInit函数中，通常应该作的事情包括: 创建至少一个窗口实例(实例是C++中的关键词)，
对传入的命令行参数进行解析，为应用程序进行数据设置和其它的一些初始化的操作.如果这
个函数返回真，wxWidgets将开始事件循环用来接收用户输入并且在必要的情况下处理这些输入。
如果OnInit函数返回假，wxWidgets将会释放它内部已经分配的资源，然后结束整个程序的运行。

 

第四个问题：Frame类是什么类？

写过java图形界面的估计也能知道的差不多了，一个Frame窗口是一个可以容纳别的窗口的顶层窗口，
通常拥有一个标题栏和一个菜单栏。来看看代码：
class MyFrame : public wxFrame
{
public:
MyFrame(const wxString& title, const wxPoint& pos, const wxSize& size);
private:
void OnHello(wxCommandEvent& event);
void OnExit(wxCommandEvent& event);
void OnAbout(wxCommandEvent& event);
wxDECLARE_EVENT_TABLE();
};
写过java的估计一看也能明白，MyFrame是构造方法，OnExit是退出的方法，OnAbout是关于的方法；
这三个估计都是Frame必备的一些方法；而OnHello就是我们自己打印HelloWorld的方法了。

第五个问题：下面这段代码是搞神马的？

wxBEGIN_EVENT_TABLE(MyFrame, wxFrame)
EVT_MENU(ID_Hello, MyFrame::OnHello)
EVT_MENU(wxID_EXIT, MyFrame::OnExit)
EVT_MENU(wxID_ABOUT, MyFrame::OnAbout)
wxEND_EVENT_TABLE()
wxIMPLEMENT_APP(MyApp);

实际上这是一个事件表，所谓事件表，是一组位于类的实现文件（.cpp文件）中的宏，用来告诉
wxWidgets来自用户或者其它地方的事件应该怎样和类的成员函数对应起来。前面展示的事件表表
明，要把标识符分别为wxID EXIT和wxID ABOUT的菜单事件和MyFrame的成员函数OnAbout和OnQuit
关联起来。这里的EVT MENU宏只是很多中事件宏中的一个，事件宏的作用是告诉wxWidgets哪种事
件应该被关联到哪个成员函数。这里的两个标识wxID ABOUT和wxID EXIT是wxWidgets预定义的宏，
通常你应该通过枚举，常量或者宏定义的方式定义你自己的标识符，比如ID_Hello。
类比于java给按钮添加监听器，这个事件表其实就是为方法添加事件响应。比如当用户点击退出菜
单项的时候，MyFrame::OnQuit函数被调用（你已经意识到了，这是事件表的功劳）。它调用wxFrame
类的Close函数来释放frame窗口。因为没有别的窗口存在了，所以就触发了应用程序的退出，实际上，
wxFrame类的Close函数并不直接关闭frame窗口，而是产生一个wxEVT CLOSE WINDOW事件，这个事件默
认的处理函数调用wxWindow::Destroy函数释放了frame窗口。

第六个问题：Frame的构造函数这么长，都什么意思啊？

首先，构造方法后面又加了: wxFrame(NULL, wxID_ANY, title, pos, size)对于我这种C++初学者就很
难理解，其实这个构造函数首先调用它的基类（wxFrame）的构造函数，使用的参数是父窗口(还没有父窗
口，所以用NULL),窗口标识(wxID ANY标识让wxWidgets自己选择一个)和标题。这个基类的构造函数
才真正创建了一个窗口的实例。除了这样的调用方法，还有另外一种方法是直接在构造函数里面显
式调用基类默认的构造函数(就是指不带任何参数的构造函数)，然后调用wxFrame::Create函数来创
建一个frame窗口的实例。理解了这个以后，里面的内容就相对简单了，无非就是添加组建了，这个
跟java编程就大同小异了，所以不用过多赘述了。

##
